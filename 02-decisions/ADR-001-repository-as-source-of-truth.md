# ADR-001: This Repository Is the Source of Truth

- **Status:** ✅ Accepted
- **Date:** 2026-05-10
- **Decision-makers:** @zhongyuxi98-jpg (interim steward), Claude (advisor)
- **Supersedes:** none
- **Related:** organization.md, governance.md, data-charter.md

---

## Context

Knowledge Gate's documentation, decisions, and configuration were spread across:

- Many separate Lovable projects (each with its own internal context, no shared state)
- Many ephemeral Claude conversations (lost when a chat ended)
- Local Word documents and DOCX templates
- The contributor's memory

Symptoms:

1. The same decision had to be re-explained to Claude every time.
2. Lovable AI agents working on different gateways made inconsistent assumptions.
3. There was no public, reviewable record of governance.
4. "Open-source / transparent governance" had no actual artifact backing it.
5. Lessons generated by Claude vanished if not manually pasted somewhere.

---

## Decision

We establish a single, public, version-controlled GitHub repository — `knowledge-gate-spec` — as the **single source of truth** for:

- Charter, governance, incentives, data charter (everything in `00-charter/`)
- Architecture overview and registries (`01-architecture/`)
- Decision records (this folder, `02-decisions/`)
- Per-gateway and per-plugin documentation
- Templates and AI agent prompts

All other surfaces — Lovable projects, Notion pages, Supabase rows, Claude prompts — must read from or sync to this repository, not the other way around.

License: CC0 1.0 Universal, so future forks and derivative organizations cannot be held back by IP claims.

---

## Alternatives considered

### A. Notion as the source of truth
- ✅ Easier to write
- ❌ Not version-controlled in a developer-friendly way
- ❌ Not directly readable by Lovable / Claude as plain files
- ❌ Cannot host a CC0 public charter in the same place as collaborative editing

### B. Lovable's "Knowledge" feature inside one project
- ❌ Locked into one project, not shared across gateways
- ❌ Not public; cannot serve as a transparent governance artifact
- ❌ Cannot be forked

### C. Self-hosted wiki (Outline / Wiki.js)
- ✅ Full control
- ❌ Requires hosting effort
- ❌ Adds infrastructure that has nothing to do with the actual mission

### D. **GitHub repository** (chosen)
- ✅ Already free, already public-by-default
- ✅ Native version control + change history
- ✅ Pull-request workflow matches the charter's "any change requires PR + review"
- ✅ Auto-syncs with Lovable projects via Lovable's GitHub integration
- ✅ Readable by AI agents as plain Markdown
- ✅ A public-domain license (CC0) is natural to apply
- ✅ Free issue tracker for governance discussions
- ⚠️ Slight learning curve for non-developer contributors → mitigated by editing on github.com directly, no command line needed

---

## Consequences

### Positive
- Every decision now has a permanent, addressable URL.
- AI agents can be pointed at this repo and immediately have correct context.
- "Open-source governance" stops being a slogan and becomes a working artifact.
- Charter and code can evolve in lockstep through the same PR review process.

### Negative / costs
- Founder must maintain discipline of updating this repo on every decision.
- Non-developer contributors will need a one-page "how to edit on GitHub" guide.
- Migrating existing scattered notes is manual work.

### Mitigations
- Add "Update knowledge-gate-spec" to every gateway's PR template.
- Reward documentation work explicitly under the contribution-points system in `incentives.md`.
